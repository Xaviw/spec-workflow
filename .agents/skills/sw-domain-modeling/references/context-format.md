# CONTEXT.md 格式

`CONTEXT.md` 位于工作流仓库根目录，是工作流原生 Skill 的项目级必读入口。专业术语和关键决策索引由 `project/memory.json` 驱动并通过 `memory` CLI 更新，不直接编辑受管块。

```markdown
# {项目名称}

{一到两句话说明项目目标和存在理由。}

## 专业术语

<!-- spec-driven:terms:start -->
**订单**：
{一到两句话定义它是什么。}
_避免使用_：交易单、购买记录
<!-- spec-driven:terms:end -->

## 关键决策

<!-- spec-driven:adr-index:start -->
- [ADR-0001 服务端决定身份归属](./adr/0001-server-owned-identity.md) — accepted · backend/auth · 客户端不得声明用户或租户归属。
<!-- spec-driven:adr-index:end -->
```

## 专业术语规则

- 多个词表达同一概念时，只保留一个规范名称，并把其他名称列入 `_避免使用_`。
- 定义说明“它是什么”，保持一到两句话；行为规则、实现方式和验收要求写到对应任务文档。
- 只收录该项目特有且会反复使用的概念；超时、错误类型、工具函数等通用技术概念不属于专业术语。
- 一词多义时拆成不同术语，不用一段宽泛定义掩盖差异。
- 自然形成多个主题时可以增加三级标题分组，但不要为尚不存在的内容预建空结构。

## 关键决策索引规则

- 每个当前或历史 ADR 都有且只有一个索引项，按编号升序排列。
- 索引项包含链接、状态、影响范围和一句话决定摘要。摘要应足以让读者判断是否需要打开正文。
- `accepted` 表示当前有效；被替代时写 `superseded by ADR-NNNN`，并保留原索引项。
- 索引是必读内容；ADR 正文只在当前任务命中其项目、仓库或模块范围时读取。
- 阶段文档依赖术语或决定时，以 inline code 写出 `CONTEXT.md` 或具体 `adr/0001-example.md` 路径，使 checkpoint 记录内容 hash。
- 首个 ADR 出现时才由 CLI 创建根目录 `adr/`，不要为空目录增加占位文件。
