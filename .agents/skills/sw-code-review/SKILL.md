# 代码审查

## 上下文契约

必读：目标仓库 Git 状态、基线到当前的 diff、提交列表和适用的仓库规范。

按需读取：`AGENTS.md` 规范路由命中的文档；工作流任务的 `prd.md`、`technical-design.md`、`spec.md`；简单任务的用户要求；被修改符号的调用方和测试。

初始禁止：无关任务、完整项目知识、与 diff 无关的全仓库扫描、默认删除计划。

输出：发现项优先的审查报告、覆盖范围、未覆盖项和残余风险。不得直接修改代码。

## 预检

1. 从 `task status --json` 的 delivery 读取每个仓库基线；简单变更则使用现场记录的基线。核对 branch/HEAD、暂存和未暂存 diff，排除初始脏文件及无关用户修改。完成：每个仓库都有明确审查范围；diff 为空时直接说明范围。
2. 按仓库和模块阅读 diff，搜索被修改符号的关键调用方、测试和契约，优先覆盖认证、权限、资金、数据写入、并发和外部网络。完成：已追到受影响边界；大 diff 已分批检查。
3. 按仓库规范和需求规格两条轴检查，并依证据定级。完成：每项发现都可复现且包含位置、条件、影响、证据和最小修复建议；没有发现时写明覆盖范围和残余风险。

## 两条审查轴

### 仓库规范

检查目标仓库明确记录的规则、现有边界和局部模式。仓库规则优先于通用气味；跳过格式化器或静态工具已可靠强制的纯样式问题。

只在真实增加风险时报告重复、职责混杂、臆测通用性、错误抽象和散弹修改，不机械套用 SOLID。

只检查与 diff 实际相关的规则。规则 ID 能提高定位精度时在发现项中引用；仓库规则与根目录规范冲突时，检查 spec 对应步骤是否记录并随实施方案确认，不擅自决定优先级。

### 需求规格

检查遗漏验收项、实现错误、范围蔓延、与 PRD/技术方案/spec 不一致，以及简单任务中用户未要求的行为。每项引用规格来源。

## 严重程度

- `P0`：可利用安全问题、密钥泄漏、数据丢失、确定性严重正确性问题。必须阻塞提交。
- `P1`：重要逻辑错误、权限遗漏、明显回归或高概率事故。默认阻塞；用户可明确接受并记录风险。
- `P2`：可维护性、局部性能或边界风险。不阻塞提交。
- `P3`：低影响命名或建议。不阻塞提交。

每项必须包含 `repo/path:line`、触发条件、影响、证据和最小修复建议。没有发现时明确说明检查过什么及残余风险。

输出发现项优先的报告后停止，不直接修改代码；由调用本 Skill 的流程按门禁处理发现。

## 修复权限

严重级别只决定门禁，不授权修改。只有任务自有、来源明确、局部可逆、不改变外部行为且不涉及密钥、权限、数据、公共契约或用户既有修改的问题，才可在用户已授权实施的范围内自动修复。

发现疑似密钥时不回显、不删除、不改历史、不自动轮换。只说明位置和风险，让用户选择迁移到本地配置、删除、忽略或轮换。
