# 形成当前 PRD

## 上下文契约

必读：当前任务的 `task.json`、`prd.md`、`decisions.md` 和用户原始需求。

按需读取：`AGENTS.md` 规范路由命中的文档、直接相关的项目索引、仓库说明、一个已有任务 PRD、为回答事实问题所需的少量源码或文档。

初始禁止：技术方案、实施方案、全部项目知识、无关任务、全仓库扫描。

输出：保持当前事实的 `prd.md`、追加式 `decisions.md`、必要的项目长期记忆更新，以及待用户确认的需求摘要。

## 流程

1. 运行 `task status <task> --json`，确认任务仍在 `prd` 且 revision 未变化；整理用户原始需求、已有 PRD、决策记录和直接相关的项目事实。完成：访谈主题、已知事实和当前需求变化已明确。
2. 调用 `grilling`，在当前请求中明确写出访谈主题、已知事实、待决范围和完成标准，并让后两者覆盖目标、用户、场景、范围、非目标、业务规则、异常、兼容、验收、约束和依赖。遇到含糊、一词多义或与 `CONTEXT.md` 冲突的表达时调用 `sw-domain-modeling`，把已经确认的项目概念同步为专业术语。完成：全部需求分支已确认或明确阻塞，关键表达与项目专业术语一致。
3. 更新唯一的 `prd.md` 和追加式 `decisions.md`，不因惯例补造业务事实；同步本轮已经确认的长期记忆变化。在“约束与依赖”中用 inline code 写出本阶段实际读取的 `CONTEXT.md`、`adr/*.md` 或 `project/*.md` 根相对路径。完成：文档反映当前需求，全部验收标准可观察且使用稳定 AC ID，长期记忆与需求结论一致，checkpoint 可追踪实际依赖。
4. 运行 `task validate <task>`。完成：校验通过、没有阻塞未决项，并已向用户展示关键结论；只有用户明确确认后，才以最新 revision 进入 `technical_design` 并调用 `sw-technical-design`。

## 文档规则

保留 CLI 生成的全部章节，只维护一份当前 `prd.md`。将原始需求与 grilling 结论合并成可读文本；需求变化时直接更新，历史交给 Git。

为每条验收标准分配稳定 ID，从 `AC-001` 开始。优先使用三级标题并写出可观察结果，例如：

```markdown
### AC-001 <可验证行为>

<前置条件、动作和预期结果>
```

不要在后续阶段重编号已确认 ID；删除或替代时在 `decisions.md` 记录原因。

## 决策记录

仅把影响范围、行为、兼容性或验收的关键选择追加到 `decisions.md`。每项写日期、决定、理由、影响；被替代时追加“取代哪项”，不改写旧记录。任务决定只有通过 `sw-domain-modeling` 的 ADR 三项门槛并取得用户确认后，才提升为项目级关键决策。

阶段命令：`task phase <task> technical_design --expected-revision <n> --confirmed`。
