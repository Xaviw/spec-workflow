# 形成当前 PRD

## 上下文契约

必读：当前任务的 `task.json`、`prd.md`、`decisions.md` 和用户原始需求。

按需读取：`AGENTS.md` 规范路由命中的文档、直接相关的项目索引、仓库说明、一个已有任务 PRD、为回答事实问题所需的少量源码或文档。

初始禁止：技术方案、实施方案、全部项目知识、无关任务、全仓库扫描。

输出：保持当前事实的 `prd.md`、追加式 `decisions.md`，以及待用户确认的需求摘要。

## 流程

1. 运行 `task status <task> --json`，确认任务仍在 `prd` 且 revision 未变化；区分事实问题和产品决策，先调查能从代码、配置或文档查明的事实。完成：问题清单只剩需要用户决定的事项。
2. 每轮批量询问当前前沿上互不依赖的问题，为每题给出推荐项、影响和选项；有依赖、高风险或回答冲突时逐题询问。覆盖目标、用户、场景、范围、非目标、业务规则、异常、兼容、验收、约束和依赖。完成：全部问题已回答或明确列为阻塞。
3. 更新唯一的 `prd.md` 和追加式 `decisions.md`，不因惯例补造业务事实。完成：文档反映当前需求，全部验收标准可观察且使用稳定 AC ID。
4. 运行 `task validate <task>`。完成：校验通过、没有阻塞未决项，并已向用户展示关键结论；只有用户明确确认后，才以最新 revision 进入 `technical_design` 并调用 `sw-technical-design`。

## 文档规则

保留 CLI 生成的全部章节，只维护一份当前 `prd.md`。将原始需求与 grilling 结论合并成可读文本；需求变化时直接更新，历史交给 Git。

为每条验收标准分配稳定 ID，从 `AC-001` 开始。优先使用三级标题并写出可观察结果，例如：

```markdown
### AC-001 <可验证行为>

<前置条件、动作和预期结果>
```

不要在后续阶段重编号已确认 ID；删除或替代时在 `decisions.md` 记录原因。

## 决策记录

仅把影响范围、行为、兼容性或验收的关键选择追加到 `decisions.md`。每项写日期、决定、理由、影响；被替代时追加“取代哪项”，不改写旧记录。

阶段命令：`task phase <task> technical_design --expected-revision <n> --confirmed`。
