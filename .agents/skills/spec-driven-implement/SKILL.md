# 实施任务

## 上下文契约

必读：`task.json`、`spec.md`、当前 slice（若有）、涉及仓库的说明和目标文件。

按需读取：`prd.md`、`technical-design.md` 的被引用章节、相关测试、DDL artifact、一个项目知识文档。

初始禁止：其他任务、全部设计文档、无关仓库、未被 spec 引用的研究材料。

输出：符合 spec 的代码和测试、及时更新的任务文档、项目事实同步、代码审查和验证入口。

## 开始前

1. 对每个涉及仓库记录 branch、HEAD 和脏文件，并与 `spec.md` 基线核对。
2. 不自动 stash、reset、checkout 或建分支。既有修改与本任务重叠时停止并让用户决定；不重叠时取得确认后继续，并在审查和提交中排除。
3. 检查前置仓库提交、依赖版本、配置和环境权限。缺失的远程写权限不应通过临时扩大权限解决。

## 实施

1. 严格按仓库顺序和 slice 执行。一次只把一个 slice 标记为 `in_progress`。
2. 使用目标仓库已有模式和依赖，避免为单次需求新增抽象或框架。
3. 可稳定隔离的业务逻辑、转换、校验和 Bug 先写能失败的测试，再实现。UI 变更验证主要状态、错误态、加载态和目标尺寸。
4. 不执行 DDL、部署、生产写入或不可逆命令；只生成命令、脚本和验证方案。
5. 发现 PRD、技术方案或 spec 与真实情况不一致时停止相关代码，先更新当前文档和 `decisions.md`，必要时请用户重新确认受影响阶段。
6. 每完成一个可验证增量立即更新 spec、slice 状态和已失真的 PRD/技术方案，不等到收尾补文档。
7. 实现检查通过后，再将验证过的长期事实同步到 `project/` 文档。

## 收尾

所有 slice 完成后调用 `spec-driven-code-review`。处理阻塞发现并再次审查；用户确认实施结果后，运行 `task phase <task> verification --confirmed`，再调用 `spec-driven-verify`。

实施阶段不自行提交。验证完成后统一给出跨仓库提交顺序、commit 内容和工作流文档提交计划，只请求一次用户确认。
