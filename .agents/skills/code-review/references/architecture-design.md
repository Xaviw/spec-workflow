# 架构与设计审查

只报告会增加当前或已知后续修改风险的设计问题。仓库既有规则和明确决定优先；SOLID 与代码气味只是判断提示，不是违规清单。

## 边界与职责

- 检查一个模块是否混合业务规则、传输、持久化或展示等独立变化原因；按职责而不是文件长度判断内聚性。
- 检查高层策略是否直接依赖低层 I/O、存储或网络细节，导致替换实现必须修改业务逻辑。
- 检查公共契约是否泄漏内部表示、允许非法状态，或迫使调用点重复相同的导航、转换和校验。

## 扩展与替换

- 当新增变体需要修改多处相同条件分派时，检查是否存在重复决策；只有真实的第二种变化已经出现时才建议抽象。
- 检查子类或实现类是否加强前置条件、削弱后置条件、拒绝父契约，或通过类型判断才能安全调用。
- 检查接口是否迫使实现提供空方法、无关依赖或占位行为；调用点是否只需要更窄的能力。
- 优先组合与清晰数据边界，不为假设需求增加参数、钩子、层级或中间转发。

## 变更形态

- 检查同一逻辑是否在多个变更块重复，或同一组参数是否持续共同传递并缺少稳定概念。
- 检查一个逻辑变化是否迫使无关文件散弹修改，或一个文件是否同时承担多个无关变化。
- 检查新增代码是否不可达、被关闭的功能路径永久遮蔽，或只服务于没有需求依据的未来场景。
- 提出重构时说明它如何改善内聚、耦合或契约，并给出保持行为的最小步骤；大型重写不作为默认建议。
