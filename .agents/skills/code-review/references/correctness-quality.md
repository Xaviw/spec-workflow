# 正确性与代码质量审查

从真实输入、失败路径和数据规模验证行为。模式只用于提出问题；没有可观察后果或契约证据时不形成发现。

## 失败处理

- 检查异常是否被吞掉、仅记录后继续、被过宽捕获或转换后丢失原因；调用者必须能区分成功、可恢复失败和终止失败。
- 检查 Promise、任务、回调、流和后台工作中的错误是否传播到正确边界，清理动作是否在失败和取消时执行。
- 检查错误消息是否既保留必要诊断上下文，又不向用户暴露堆栈、查询、路径或敏感数据。
- 检查解析、网络、文件、数据库和序列化失败是否具有与调用方契约一致的回退或停止行为。

## 输入与边界

- 检查 `null`、缺失值、`0`、`false`、空字符串和仅空白字符串是否被错误地合并处理。
- 检查空集合、首尾元素、分页边界、负数、零、最大值、除零、溢出、浮点比较和 off-by-one。
- 检查字符串长度、Unicode、组合字符和规范化是否影响验证、截断、计数、比较或显示。
- 检查非法状态是否能越过入口验证，并在更深层以不一致形式失败。

## 数据访问与性能

- 检查循环内查询或远程调用、缺少批处理、无分页读取、过度查询和依赖缺失索引的新增访问模式。
- 检查高频路径中的重复解析、正则编译、序列化、加密和可避免计算；以真实调用频率和数据规模判断影响。
- 检查集合、缓冲区、队列和缓存是否无界增长；大文件或大结果是否应流式处理。
- 检查缓存的 TTL、失效、键唯一性和用户或租户隔离，避免长期陈旧、键碰撞和跨用户数据复用。

## 测试证据

- 检查测试是否覆盖本次行为的成功、失败和关键边界，而不只是执行代码或重复实现细节。
- 检查新增测试在缺少本次修复时是否会失败；不能证明行为的快照或宽泛断言不作为充分证据。
- 并发、事务、缓存和外部系统行为只有在确定性测试或等价证据支持时才视为已覆盖。
