# 代码审查

## 上下文契约

必读：当前请求已明确的审查范围；未明确时读取当前仓库的 Git 状态、暂存与未暂存 diff，以及未跟踪文件内容。

按需读取：范围内适用的仓库说明、需求或规格、架构决定、被修改符号的调用点、测试、公共契约和下述专项参考；工作流带 task 调用时读取调用方提供的项目入口和任务显式引用的项目记忆。

初始禁止：无关任务、与范围无关的全仓库扫描、默认删除计划和推测性重构。

输出：发现项优先的审查报告、实际覆盖范围、未覆盖项和残余风险；不得修改代码。

## 流程

1. 确定并固定范围。优先采用当前请求明确的仓库、路径、符号、diff、提交范围、包含项和排除项；文件或路径只限制目标，只有明确要求完整审查或审计时才检查完整内容。提交或分支审查只使用请求中明确的比较基线，缺少基线时先确认。没有明确范围时审查当前仓库全部暂存、未暂存和未跟踪变更；没有变更时说明无可审查范围并停止。完成：已确定仓库、审查模式、基线与终点、包含项和排除项，后续查证不扩大范围。
2. 建立证据上下文。读取范围适用的仓库规则；有需求或规格时逐项对照；追踪被修改符号的关键调用点、测试和公共契约。工作流调用只使用调用方按 task 或已登记 repo/module 范围筛选出的专业术语、ADR 和项目事实，不自行扫描整个工作流记忆；独立调用不要求存在这些文件。大范围按逻辑模块分批。仓库明确规则和 accepted ADR 优先；工具已可靠强制的纯样式不再人工检查。完成：每个范围内变更均已检查，关键行为已追到受影响边界，适用项目决定没有遗漏。
3. 按变更特征读取专项参考，只加载直接相关的文件：
   - 涉及外部输入、身份、权限、租户、敏感信息、文件、网络、加密、依赖、数据写入、事务、并发、缓存或事件时，读取 [安全与可靠性](references/security-reliability.md)。
   - 涉及失败处理、异步操作、输入校验、集合或数值边界、查询、批处理、缓存、CPU、内存或 I/O 时，读取 [正确性与代码质量](references/correctness-quality.md)。
   - 涉及公共接口、继承、实现类、共享抽象、模块边界、条件分派或结构重构时，读取 [架构与设计](references/architecture-design.md)。
   完成：适用参考已读取，不适用参考未进入上下文。
4. 检查三个维度：实现行为是否正确且符合已知意图；安全、可靠性、边界和性能是否会在真实条件下失效；架构、仓库规范和测试是否支持当前变更。没有需求来源时仍检查代码自身正确性，并把需求符合性列为未覆盖。完成：所有适用维度均已检查。
5. 验证、去重并定级。发现必须由范围内代码直接引入或暴露，具有具体触发条件和可观察影响；范围外旧问题只在阻碍判断时列为残余风险。同一根因只报告一次，证据不足的怀疑不列为发现。完成：报告中没有纯偏好、推测或重复项。

## 严重程度

- `P0`：可直接利用的严重安全问题、凭据泄漏、不可逆数据丢失或系统性不可用。
- `P1`：重要逻辑错误、权限或数据隔离遗漏、数据一致性问题、明显回归或高概率生产事故。
- `P2`：影响受限的行为缺陷、可靠性或性能风险、会显著增加后续修改风险的设计问题。
- `P3`：具有具体后果的低影响问题；仅在用户要求完整建议时报告，纯风格和主观偏好始终忽略。

严重级别只描述风险，不决定工作流门禁，也不授权修改。

## 报告

按严重程度列出发现。每项包含类别、相关位置、触发条件、影响、证据和最小修复建议；行为缺失可以引用规格位置及最接近的实现入口，不得虚构代码行。发现之后列出实际范围、依据、未覆盖项和残余风险。没有发现时明确说明“未发现问题”，但仍保留覆盖范围和残余风险；测试通过不能替代审查结论。

发现疑似密钥时不回显内容，只报告位置和风险，不删除、不改历史、不自动轮换。
