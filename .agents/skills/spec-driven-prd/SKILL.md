# 形成当前 PRD

## 上下文契约

必读：当前任务的 `task.json`、`prd.md`、`decisions.md` 和用户原始需求。

按需读取：直接相关的项目索引、仓库说明、一个已有任务 PRD、为回答事实问题所需的少量源码或文档。

初始禁止：技术方案、实施方案、全部项目知识、无关任务、全仓库扫描。

输出：保持当前事实的 `prd.md`、追加式 `decisions.md`，以及待用户确认的需求摘要。

## Grilling

1. 先区分事实问题和产品决策。能从代码、配置或文档查明的事实先调查，不拿来询问用户。
2. 每轮列出当前前沿上互不依赖的决策问题；为每题给出推荐项、影响和可选项，让用户批量回答。
3. 存在依赖、较高风险或回答冲突时，退回逐题询问。
4. 重点澄清：目标、用户、场景、范围、非目标、业务规则、异常路径、兼容性、验收标准、约束和依赖。
5. 不因“通常如此”补造业务事实。未决问题要么继续询问，要么明确保留为阻塞项。

## PRD 结构

`prd.md` 只保留一份当前版本，至少包含：

```markdown
# <任务标题>
## 背景与原始需求
## 目标
## 非目标
## 用户与场景
## 范围和业务规则
## 异常与边界
## 验收标准
## 约束与依赖
## 未决问题
```

将原始需求与 grilling 结论综合成可读文本，不另存原始需求副本。需求改变时直接更新当前 PRD；Git 负责保留历史。

为每条验收标准分配稳定 ID，从 `AC-001` 开始。优先使用三级标题并写出可观察结果，例如：

```markdown
### AC-001 <可验证行为>

<前置条件、动作和预期结果>
```

不要在后续阶段重编号已确认 ID；删除或替代时在 `decisions.md` 记录原因。

## 决策记录

仅把影响范围、行为、兼容性或验收的关键选择追加到 `decisions.md`。每项写日期、决定、理由、影响；被替代时追加“取代哪项”，不改写旧记录。

完成后运行 `task validate <task>`，逐条检查验收标准是否可验证。向用户展示关键结论和未决问题；只有用户明确确认 PRD，才使用 `task status` 返回的 revision 运行 `task phase <task> technical_design --expected-revision <n> --confirmed`。
