# 编写高质量技能

Skill 的核心目标是让随机系统稳定采用同一套过程，而不是每次生成相同结果。所有内容都应服务于这种可预测性。

遇到不熟悉的粗体术语或需要诊断问题时，读取 [GLOSSARY.md](GLOSSARY.md)。

## 编写流程

1. 明确调用方式。
   - 仅用户调用：在 `agents/openai.yaml` 中设置 `policy.allow_implicit_invocation: false`。
   - 允许模型调用：保留默认策略，并让 `description` 同时说明能力和真正独立的触发分支。
   - 本工作流的 `SKILL.md` frontmatter 只保留 `name` 和 `description`。
2. 列出不同调用分支。正文内联每个分支都需要的内容，只把分支专用资料放入通过明确条件链接的参考文件。
3. 按信息层级组织内容：先放有顺序的步骤，再放执行时随时需要的规则，最后放按需读取的外部参考。
4. 为每个步骤写可检查的完成条件。条件应清楚说明何时完成；需要完整覆盖时，用“每个”“全部”等可核对范围约束结果。
5. 用模型已熟悉的引导词压缩反复出现的行为原则，例如“近端发展区”“红灯测试”或“渐进披露”。优先使用已有概念，不创造必须额外解释的新术语。
6. 逐句精简并验证：每个意思只保留一个权威位置；删除失效、重复或不改变模型行为的句子；运行技能校验及最小的真实调用检查。

完成标准：调用策略与用户要求一致；每个步骤都有清晰边界；每个分支只加载所需上下文；正文无占位、重复和无效指令；技能校验通过。

## 调用描述

允许模型调用时，`description` 是常驻上下文中的触发入口：

- 先写最能唤起该能力的引导词，再写触发条件。
- 每个独立分支只写一个触发条件；同义改写不是新分支。
- 删除正文已有的身份介绍和实现细节。
- 只有当模型或其他 Skill 必须主动找到它时，才支付隐式调用的上下文成本。

仅用户调用时，描述保持一行人类可读摘要，并明确“仅当用户明确调用 `$skill-name` 时使用”。不要把这类 Skill 接入自动路由。

## 拆分与渐进披露

- 当一个概念需要独立触发时，才按调用拆成新 Skill。
- 当后续步骤会诱使模型过早结束当前步骤，且完成条件无法再明确时，才按顺序拆分。
- 只有部分分支需要的资料放入外部参考；所有分支都需要的规则留在正文。
- 同一概念的定义、规则和例外放在一起，避免读到一半还需跨文件拼接。

## 常见失效

- **过早完成**：先收紧当前步骤的完成条件；仍然会抢跑时，再隐藏后续步骤。
- **重复**：保留单一事实来源，用一个引导词代替多处同义解释。
- **沉积**：定期删除已失效或偏离技能目标的内容。
- **蔓延**：将分支专用参考移出正文，必要时再按真实分支拆分。
- **无效指令**：删除模型默认就会执行、且不能改变行为的句子。
- **否定引导**：优先描述目标行为；确需硬性禁令时，同时给出应该采取的正向动作。
