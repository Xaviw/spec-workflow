# 路由开发请求

## 上下文契约

必读：`AGENTS.md`、`AGENTS.local.md`、用户请求。

按需读取：`node tools/workflow.js task candidates --json` 返回的可用任务元数据；只有 AI 初判可能相关时才读取对应 `prd.md`。

初始禁止：所有任务正文、全部迭代、全部项目知识、目标仓库源码。

输出：任务分类、最多 5 个相关任务候选及一个合并后的用户选择问题。

## 流程

1. 检查用户是否已给出任务 ID、路径、链接、唯一名称，或本轮已选定唯一任务。若是，读取其 `task.json`，运行 `task status <task> --json` 并按阶段路由，不搜索候选。完成：已交给唯一的阶段 Skill。
2. 否则同时判断任务规模和相关任务。使用 `CONTEXT.md` 的专业术语理解请求，运行 `task candidates --json`，先按以下硬门禁分类，再从 CLI 候选中筛选最多 5 个真正相关项。完成：得到一个分类结论和 0-5 个候选。
3. 合并成一次用户选择；用户选定任务后立即运行 `task status <task> --json`，选定新任务时同时确定已有开放迭代或新建迭代。完成：进入 `sw-simple-change`、唯一现有任务的阶段 Skill，或已具备创建工作流任务所需的迭代选择。

不持久化活动任务；新会话中的未提交目录不代表用户已选择任务。

## 分类门禁

仅当以下条件全部满足时成立：

- 要求和验证方式清楚；
- 只涉及一个仓库中的局部模块；
- 一次上下文可完成；
- 不涉及数据库结构或迁移、安全或权限、公共契约、生产环境、不可逆操作。

不满足任一项则推荐工作流任务。用户可对工作量、跨仓库或一般不确定性选择降级，但不得绕过硬门禁。

## 候选判断

运行候选命令。CLI 只做确定性过滤：保留开放迭代中未取消，且未提交或最近 30 天有 Git 更新的任务；返回路径、标题、摘要、阶段、仓库、模块、来源、创建者邮箱和 Git 更新时间。未提交任务排在前面，其余按更新时间排列。

CLI 不接收需求文本，不做语义相关性或 Top 5 截断。根据用户请求判断相关性；作者邮箱只作线索，元数据不足时只读取疑似相关任务的 PRD。不得自动关联。

## 选择分支

- 简单任务且存在相关任务：询问“同步到候选任务”或“作为独立简单变更”。
- 工作流任务且存在相关任务：询问“继续候选任务”或“新建任务”。
- 新建任务：在同一轮要求选择已有开放迭代或新建迭代。
- 没有候选：简短说明分类结论；简单任务可直接继续，工作流任务要求选择迭代。

若用户选择同步到工作流任务，必须遵守该任务当前阶段，不得走简单任务捷径。

用户要求取消任务时，先运行 `task delete <task>` 预览安全删除资格，再询问“删除”或“保留记录并标记取消”。只有用户选择后才执行 `task delete --apply` 或 `task cancel --reason ... --confirmed`；不得主动标记取消。
