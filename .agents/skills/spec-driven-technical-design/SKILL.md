# 编写技术方案

## 上下文契约

必读：`task.json`、已确认的 `prd.md`、`decisions.md`、涉及仓库的说明文档。

按需读取：目标模块源码和测试、数据库规范、API 契约、配置说明、项目知识。每次读取都必须服务于一个明确的设计问题。

初始禁止：其他任务正文、无关仓库、完整项目知识库、实施步骤细节。

输出：`technical-design.md`，必要时产生 `artifacts/ddl/*.sql` 或 `research/*.md`。

## 证据优先

1. 扫描实际入口、调用链、数据模型、复用点和边界。关键证据记录为 `repo_id / path / symbol / line`。
2. 区分当前事实与目标事实。当前代码不能否定已确认 PRD；差异必须写成待实现或迁移项。
3. 复杂调查可暂存 `research/`。被方案引用的证据保留，可复用事实在验证后提升到 `project/knowledge/`，其余噪声删除。

## 固定影响扫描

每项都给出设计或“不适用”的理由：

- 数据库和数据迁移；
- API、事件、SDK 等公共契约；
- 认证、授权、租户、隐私与安全；
- 配置、环境变量、配置中心和外部服务；
- 后端、Web 或小程序 UI 状态；
- 日志、指标、告警和审计；
- 跨仓库依赖、版本和实施顺序；
- 兼容、灰度、发布、回滚和验证。

## 数据库要求

涉及结构或数据迁移时必须提供：当前 schema 证据、目标 DDL、迁移或回填、锁与容量风险、执行顺序、回滚方式和验证 SQL。

短且单步的 DDL 放在正文；复杂或多步 DDL 写入 `artifacts/ddl/*.sql` 并由正文引用。DDL 仅作为方案产物，绝不执行。没有项目数据库规范时，不擅自发明命名或索引规则，将其列为待确认项。

## 文档结构

```markdown
# 技术方案
## 目标与非目标
## 当前实现与证据
## 总体设计
## 数据模型与 DDL
## API 与调用方
## 后端实现
## Web/小程序实现
## 权限与安全
## 配置、环境与联调
## 可观测性
## 跨仓库依赖与顺序
## 迁移、发布与回滚
## 测试和验证策略
## 风险与未决问题
```

在正文中引用每个 PRD 验收 ID，确保所有 AC 都有设计落点，跨章节字段和状态一致。运行 `task validate <task>`；只有用户明确确认技术方案，才使用当前 revision 运行 `task phase <task> implementation_spec --expected-revision <n> --confirmed`。
