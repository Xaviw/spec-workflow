# 编写高质量技能

Skill 的目标是让模型在目标场景中稳定采用合适过程。只保留能改变模型行为、提供非显然知识或提高执行可靠性的内容。

遇到不熟悉的粗体术语或需要诊断问题时，读取 [术语表](GLOSSARY.md)。

## 流程

1. 明确目标和真实用例。确认 Skill 的存放位置、目标运行时、预期产物，以及应该和不应该触发的具体请求；更新现有 Skill 时先读取其全部文件和适用规范。完成：能力边界、触发边界和至少一个代表性用例均已明确。
2. 确定调用策略。区分仅用户显式调用、允许模型主动调用，以及由其他 Skill 明确调用；调用来源可以不同，但相同行为只维护一套流程。完成：每个独立触发分支都有必要性，调用策略可由目标运行时可靠表达。
3. 规划可复用内容。从用例反推执行步骤，以及需要复用的脚本、参考和输出资源；确定性或高风险操作优先使用可验证脚本，分支专用知识使用参考文件。完成：每项资源都有真实用途，不需要的目录和说明文件未创建。
4. 编写 Skill。遵循目标运行时的格式和命名规则；在常驻描述中说明能力与触发条件，在正文中保留所有分支共用的最小流程、边界和完成条件，并通过明确条件链接专项参考。完成：另一实例只读取当前请求和该 Skill 也能选择正确分支并执行。
5. 精简和校验。合并重复规则，删除模型默认会做且不改变行为的句子、失效说明和推测性分支；运行目标运行时提供的校验、脚本测试及最小真实调用检查。复杂 Skill 使用互不泄漏预期答案的代表性用例做前向测试。完成：格式有效、引用可达、触发不过宽、流程无矛盾，成功与不触发场景均有证据。

## 编写原则

- 常驻描述优先写“做什么”和“何时使用”；实现细节只在触发后加载。
- 按任务可变性设置自由度：开放判断使用原则，稳定模式使用参数化步骤，脆弱操作使用确定性脚本和严格验证。
- 每个步骤写可观察的完成条件；需要完整覆盖时明确“每个”“全部”等范围。
- 一个意思只保留一个权威位置。所有分支都需要的规则留在正文，只有部分分支需要的资料放入一级参考。
- 优先使用模型已熟悉的概念和正向动作；硬性边界同时说明可采取的替代动作。
- 名称简短、稳定并描述能力。是否使用命名空间取决于真实依赖，不用前缀表达手动或自动调用权限。

## 交付

报告已创建或修改的 Skill、触发策略、资源选择、验证结果和仍未覆盖的用例。除非用户要求，不为 Skill 添加 README、变更日志、安装指南或其他辅助文档。
