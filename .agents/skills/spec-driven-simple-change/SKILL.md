# 执行简单变更

## 上下文契约

必读：`AGENTS.md`、`AGENTS.local.md`、用户请求、目标仓库文档。

按需读取：相关任务（若用户选择同步）、目标模块代码、测试、`project/repositories/<repo-id>.md`、一个相关知识文档。

初始禁止：无关任务、全部项目知识、其他仓库源码、`verification.md` 工作流模板。

输出：最小代码修改、相关验证结果、code-review 结论，以及经确认后的提交和 `changes.jsonl` 记录。

## 流程

1. 确认目标仓库、局部范围、可执行验证和用于记录该变更的开放迭代。存在多个开放迭代时让用户选择，不推断“当前迭代”。再次发现硬门禁时停止，转为工作流任务。
2. 记录目标仓库当前分支、HEAD 和脏文件。与任务范围重叠的既有修改必须让用户决定；不重叠修改经确认后排除。
3. 用目标仓库现有模式完成最小修改。可稳定隔离的逻辑或 Bug 优先先写失败测试；纯 UI 或遗留代码不强制引入测试框架。
4. 即时同步因变更失真的项目文档，即使本次没有建立工作流任务。
5. 调用 `spec-driven-code-review`，再运行与变更直接相关的测试、类型检查、构建或人工检查。
6. 自动验证不能覆盖业务行为时，明确列出唯一需要用户手工确认的事项；否则不要额外打断。
7. 汇总一次跨仓库提交计划并请求一次确认。不得 push 或 merge。
8. 代码提交成功后，使用 `change add` 将摘要、各仓库 commit、验证和更新的项目文档追加到所选开放迭代的 `changes.jsonl`，再提交工作流仓库记录。

## 审查处置

P0 阻塞提交，但不授权自动修改。P1 可由用户明确接受风险；P2/P3 不阻塞。仅可自动修复任务自有、来源明确、局部可逆且不改变行为的问题；密钥、权限、数据、公共契约和来源不明修改必须由用户决定。
