# Redis 设计与使用规范

| 属性 | 值 |
| --- | --- |
| 状态 | active |
| 适用范围 | Redis 缓存、锁、队列和计数场景 |
| 来源 | Redis 设计规范，经工作流补充可靠性边界 |

## 强制规则

| ID | 规则 | 验证方式 |
| --- | --- | --- |
| `REDIS-001` | 生产实例必须启用认证和网络访问控制，凭据不得进入代码和日志 | 配置依赖与网络边界检查 |
| `REDIS-002` | Key 使用稳定业务前缀和层次分隔，不包含空白、换行或不可控用户原文 | Key 构造调用点检查 |
| `REDIS-003` | 临时、缓存和幂等 Key 必须定义 TTL；大量 Key 的过期时间应加入合理抖动 | TTL 和过期分布测试 |
| `REDIS-004` | 禁止在线请求路径使用 `KEYS` 等阻塞或高危命令 | 静态扫描和命令权限检查 |
| `REDIS-005` | 重试必须有次数上限、退避、抖动和可观测性，不得无限快速重试 | 故障注入测试 |
| `REDIS-006` | 分布式锁必须使用唯一持有者标识并原子校验释放；锁超时不得直接复用业务缓存 TTL | 并发和超时测试 |
| `REDIS-007` | 进入 Redis 的敏感数据遵循 `security.md`，不得因缓存降低授权和加密要求 | 数据抽样与权限检查 |

## 设计检查

- Redis 是缓存还是事实来源，丢失、淘汰和重启后的行为是什么？
- Key、Value 和集合元素的预估大小、增长上限和热点分布是什么？
- 缓存穿透、击穿、雪崩和数据一致性采用什么最小策略？
- 连接超时、命令超时、重试和降级是否会放大故障？
- Cluster 场景是否确实需要 hash tag，是否会制造单分片热点？

## 代码审查与验证

- 不用固定的通用数字替代容量测量；大 Key 和热 Key 阈值由实例能力及压测确定。
- 检查无 TTL Key、通配扫描、超大序列化对象和不受控集合增长。
- 验证缓存未命中、Redis 不可用、锁超时、重复请求和并发回填。
- 记录监控项：命中率、内存、淘汰、慢命令、连接、热点和复制状态。
