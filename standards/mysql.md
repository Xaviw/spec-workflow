# MySQL 设计与使用规范

| 属性 | 值 |
| --- | --- |
| 状态 | active |
| 适用范围 | 使用 MySQL 的目标仓库 |
| 来源 | 数据库设计使用规范，经工作流去除错误阈值和教程内容 |

数据库版本、SQL mode、字符集、排序规则和目标仓库既有约定必须先取证。版本不明时不得套用固定容量或索引阈值。

## 强制规则

| ID | 规则 | 验证方式 |
| --- | --- | --- |
| `DB-001` | 应用不得使用 root 或管理账号；账号权限按服务和读写场景最小化 | 配置依赖和授权清单检查 |
| `DB-002` | 新建库表显式使用项目确认的 `utf8mb4` 字符集和一致排序规则 | DDL 静态检查 |
| `DB-003` | 默认使用 InnoDB；其他引擎必须记录原因、限制和确认 | DDL 静态检查 |
| `DB-004` | 表、字段、索引必须遵循目标仓库既有命名；新建对象使用小写 snake_case 并提供注释 | schema 对比 |
| `DB-005` | 业务实体必须有稳定主键；审计字段、软删除和唯一约束按业务完整性要求设计 | DDL 与数据模型检查 |
| `DB-006` | 金额和精确比例使用整数最小单位或满足精度要求的 DECIMAL，不使用浮点数 | 边界计算和 DDL 检查 |
| `DB-007` | SQL 必须参数化；客户端字段、排序和 SQL 片段只能映射到服务端白名单 | 注入测试和代码审查 |
| `DB-008` | UPDATE、DELETE 和业务查询必须有明确范围；批量操作说明影响行数、锁、事务和恢复方式 | 执行计划及方案审查 |
| `DB-009` | 事务应保持短小，外部网络和文件调用不得无理由放在事务内 | 调用链和事务边界检查 |
| `DB-010` | 结构或数据迁移必须包含当前 schema 证据、目标 DDL、容量与锁风险、顺序、回滚和验证 SQL | 技术方案完整性检查 |
| `DB-011` | 工作流不得执行 DDL 或生产数据修改 | 只保留 artifact 和静态验证证据 |

## 设计约定

- 主键是否自增、使用分布式 ID 或自然键，由业务、分库方式和既有仓库共同决定，不设置跨项目绝对规则。
- 时间字段优先使用项目统一的 `datetime` 或 `timestamp` 语义；不得为了表达空值改用无说明的整数时间。
- 文件只保存稳定对象键或相对路径，不把临时签名 URL 和环境域名固化到业务表。
- 是否允许 NULL、外键、唯一索引、冗余字段和分区表必须基于数据完整性与访问模式决定，不使用一刀切禁令。
- 表和索引数量、批次大小、IN 列表、事务语句数、慢 SQL 阈值由版本、数据量、执行计划和压测证据确定。

## 查询与索引

- SELECT 和 INSERT 显式列出字段，避免因 schema 演进读写意外数据。
- 索引由真实查询、选择性、排序和写入成本驱动；删除冗余索引前必须确认调用方。
- 检查索引列上的函数、隐式类型转换、前导模糊匹配和深分页。
- 多表查询先用执行计划和数据量证明可接受；不得仅按 JOIN 表数量判定正确性。
- 读写分离必须处理复制延迟；写后强一致读取、事务读取和关键状态读取应有明确数据源。
- 禁止把完整 SQL 结构交给客户端控制，也不得通过拆成大量循环查询来规避一个可控查询。

## 迁移与回滚

- 大表 ALTER、回填、索引构建和分区变化必须估算锁、磁盘、复制延迟和执行窗口。
- 采用扩展再收缩的兼容顺序：先新增并兼容读写，再迁移数据和调用方，最后在独立确认后清理旧结构。
- 回滚必须说明代码、schema 和数据是否可逆；不可逆转换提供备份、校验和停止条件。
- DDL artifact 只包含待执行内容，不混入账号、密码、生产地址或人工确认口令。

## 代码审查与验证

- 审查参数绑定、权限条件、事务连接传递、错误保留和连接池配置。
- 对关键 SQL 保存脱敏后的 EXPLAIN 摘要，不伪造生产数据量结论。
- 验证空值、重复、并发、死锁重试、写后读和迁移前后数据一致性。
- verification 记录 DDL 静态审查、执行顺序、回滚和验证 SQL；明确标注“未执行 DDL”。
