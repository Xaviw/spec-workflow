本文件描述既有兼容协议，不批准新系统直接采用。未在目标仓库发现实现、测试和安全确认时，不得仅依据本文新增签名或加密逻辑。

## 待确认的现有协议

- 请求签名格式：`<timestamp>.<hmac>.<nonce>`。
- 签名输入按固定顺序包含请求体摘要、URI、Token、时间戳和 nonce。
- 请求体摘要使用 SHA-256，消息认证使用 HMAC-SHA256。
- 既有加密载荷使用 `Data` 和 `Notice` 字段；具体 AES 模式、填充和密钥派生必须以目标仓库实现为准。

## 强制规则

| ID | 规则 | 验证方式 |
| --- | --- | --- |
| `SIGN-001` | 协议版本、字段名、时间戳单位、字符编码、JSON 规范化和 URI 规则必须唯一确定 | 跨语言固定测试向量 |
| `SIGN-002` | 时间戳统一为秒级，并定义允许时钟偏差；JavaScript 实现不得直接使用毫秒值 | 过期、未来时间和边界测试 |
| `SIGN-003` | nonce 必须使用安全随机源，服务端在有效窗口内拒绝重复 nonce | 重放测试 |
| `SIGN-004` | 比较签名时使用常量时间实现；失败响应不得泄露参与计算的密钥材料 | 代码审查和失败响应检查 |
| `SIGN-005` | Token、PKey、派生密钥和解密明文不得写入日志或错误响应 | 日志与错误路径检查 |
| `SIGN-006` | 签名验证、解密、业务处理和响应保护的顺序必须由协议明确，不允许按项目口头约定 | 请求和响应时序测试 |
| `SIGN-007` | 密钥来源、权限、有效期、轮换、撤销和多版本兼容必须有可执行方案 | 配置与轮换演练 |
| `SIGN-008` | 每种支持语言至少共享一组不含真实凭据的固定测试向量 | CI 契约测试 |

## 安全复核重点

- 现有设计是否把公开标识当作 HMAC 密钥，或让客户端长期持有等价服务端密钥。
- 加密是否同时提供完整性和来源认证；旧 CBC 兼容路径如何防止篡改和填充攻击。
- GET 参数排序、重复参数、空值、百分号编码、路径前缀和 JSON 字段顺序是否确定。
- 请求和响应是否都绑定方法、路径、主体、版本和身份，是否存在跨接口重放。
- Token 获取、签名和加密失败时是否有稳定错误码及限流策略。

## 验证证据

- 安全责任方对协议版本和风险的确认记录。
- 服务端与每个客户端语言共同通过的固定测试向量。
- 时间偏差、nonce 重复、主体篡改、路径篡改、错误密钥和密钥轮换测试。
- 日志、监控和错误响应中不含密钥材料的检查结果。
