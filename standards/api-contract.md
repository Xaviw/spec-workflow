# API 契约规范

| 属性 | 值 |
| --- | --- |
| 状态 | active |
| 适用范围 | 采用公司公共接口协议的 HTTP API |
| 来源 | API 设计规范 v1.0.1、公共接口参数说明 v2，经工作流合并 |

本文件记录公司既有接口契约，不代表通用 REST 设计原则。目标仓库已有契约不同时，以仓库事实为准，并在技术方案中明确兼容策略。

## 强制规则

| ID | 规则 | 验证方式 |
| --- | --- | --- |
| `API-001` | 现有协议默认使用 POST；回调、健康检查和仓库既有契约可例外，但必须记录原因 | 契约测试和路由检查 |
| `API-002` | 路径、方法名和字段命名必须在同一 API 域内保持一致，不自创缩写或混用拼音 | OpenAPI 或接口清单检查 |
| `API-003` | 采用公共响应封装时固定包含 `Code`、`Msg`、`Data`、`Ts`、`RequestId` | 契约测试 |
| `API-004` | `RequestId` 是唯一链路字段；不得在同一协议中交替使用 `Sno` 等未声明别名 | 请求和响应样例检查 |
| `API-005` | 时间点默认使用秒级 Unix 时间戳；持续时间必须同时声明单位 | 边界值和序列化测试 |
| `API-006` | 金额和比例使用约定的最小整数单位，并在字段说明中声明单位 | 契约和计算测试 |
| `API-007` | 可能超出 JavaScript 安全整数范围的整数使用字符串传输 | 大数契约测试 |
| `API-008` | 集合无数据时返回空集合，不用 `null` 表示空集合 | 空结果测试 |
| `API-009` | 服务端必须执行字段、格式、长度、枚举和权限校验 | 无效输入与越权测试 |
| `API-010` | 已发布字段不得无计划删除、改名或改变语义；不兼容变化使用新版本 | 调用方影响扫描和兼容测试 |

## 公共响应

```json
{
  "Code": "200",
  "Msg": "操作成功",
  "Data": {},
  "Ts": 1710000000,
  "RequestId": "request-id"
}
```

- `Code`：稳定的机器可读业务码。成功值为 `200`；失败码不得依赖 `Msg` 才能区分处理方式。
- `Msg`：面向调用方的简短描述，不包含堆栈、SQL、内部路径或敏感数据。
- `Data`：当前接口声明的业务数据。
- `Ts`：秒级服务端时间戳。
- `RequestId`：用于日志关联的问题定位标识。

## 命名与分页

- 使用完整、稳定、行业可理解的英文名称；同一概念在请求、响应和不同实体中保持一致。
- 采用大驼峰字段的项目应保持现有风格；新项目是否采用该风格由仓库规范决定。
- 页码分页至少声明 `PageNo`、`PageSize` 和 `Total` 的含义；游标分页应使用不透明且稳定的游标，不把页码和游标语义混用。
- 排序字段只能映射到服务端白名单，不得把客户端字符串直接拼入 SQL。

## 设计检查

- 谁是调用方，是否存在旧客户端或跨仓库依赖？
- 变化是向后兼容、需要新版本，还是需要迁移窗口？
- 错误码、空值、时间、金额、大整数和分页边界是否明确？
- 身份和数据权限来自哪里，是否存在对象遍历风险？

## 代码审查与验证

- 对照接口文档检查路径、字段、类型、单位、错误码和权限。
- 至少覆盖成功、参数错误、未认证、无权限、资源不存在和服务异常。
- 对公共字段、空集合、大整数、时间单位和旧调用方执行契约测试。
- 公共契约变化必须在技术方案、spec、验证记录和调用方文档中保持一致。
