# 安全与敏感信息规范

| 属性 | 值 |
| --- | --- |
| 状态 | active |
| 适用范围 | 全部目标仓库 |
| 来源 | 数据安全敏感信息、配置文件管理和前端安全规范，经工作流精简 |

## 强制规则

| ID | 规则 | 验证方式 |
| --- | --- | --- |
| `SEC-001` | 代码、配置样例、测试、日志和文档不得包含明文凭据 | 密钥扫描；人工检查配置入口和示例 |
| `SEC-002` | 凭据必须来自环境变量、配置中心或密钥管理系统；只记录变量名和获取方式 | 检查配置加载链路，不读取或回显实际值 |
| `SEC-003` | 不得枚举或记录全部环境变量、请求 Header、Cookie 或进程启动参数 | 检查诊断代码和日志包装层 |
| `SEC-004` | 身份、租户、机构和数据范围必须来自服务端可信上下文，并在查询和写入时约束 | 使用越权用例验证资源访问 |
| `SEC-005` | 收集、返回、日志记录和导出个人信息时遵循最小必要原则 | 对照接口字段、日志和导出文件检查 |
| `SEC-006` | 密码不得可逆保存；使用目标平台维护的密码散列实现和独立随机盐 | 检查密码存储和验证路径 |
| `SEC-007` | 禁止自创加密、签名、随机数或密钥派生算法；使用维护中的标准库 | 检查算法、模式、密钥来源和随机数来源 |
| `SEC-008` | 面向浏览器的会话 Cookie 应按场景设置 HttpOnly、Secure 和 SameSite；状态变更请求必须防护 CSRF | 检查 Set-Cookie 和跨站请求验证 |
| `SEC-009` | 用户可控内容进入 HTML、URL、脚本、SQL、命令或模板时必须按目标上下文编码或参数化 | 使用注入和 XSS 用例验证 |

## 敏感信息分类与展示

| 分类 | 示例 | 默认处理 |
| --- | --- | --- |
| 凭据 | 口令、Token、Cookie、API Key、私钥 | 不展示、不记录 |
| 身份 | 姓名、证件号、社保卡、护照 | 最小化返回，展示时脱敏 |
| 联系方式 | 手机、固定电话、邮箱、详细地址 | 展示时脱敏 |
| 财产 | 银行卡、余额、交易明细、支付记录 | 严格授权，默认不完整展示 |
| 生物与健康 | 人脸、指纹、基因、病历 | 高敏感数据，单独评估和授权 |
| 设备与位置 | IMEI、IDFA、MAC、精确经纬度、轨迹 | 最小化收集，限制用途和保留期 |
| 商业秘密 | 合同、内部价格、企业支付信息 | 按业务角色授权和审计 |

脱敏规则必须以项目确认的数据分类为准。识别和脱敏实现应由代码及测试维护，不把易过期的超长正则作为规范正文的事实来源。

## 密码学基线

- 密码散列使用目标平台维护的 Argon2id、bcrypt、scrypt 或 PBKDF2 实现；禁止直接使用 MD5、SHA-1 或无盐快速散列保存密码。
- 新设计优先使用带认证的对称加密模式，例如 AES-GCM 或符合监管要求的 SM4-GCM。
- RSA 密钥至少 2048 位；具体算法、密钥长度和国密要求由项目安全责任方确认。
- 密钥和 nonce 使用密码学安全随机数源；密钥必须支持轮换、撤销和权限隔离。
- 公共签名或加密协议必须经过安全复核，并提供固定测试向量和兼容性说明。

## 设计检查

- 涉及哪些身份、权限、个人信息、商业秘密和监管要求？
- 信任边界、攻击入口、密钥来源和轮换方式是什么？
- 数据在传输、存储、日志、缓存、备份和导出中如何保护？
- 权限拒绝、脱敏、过期、删除和审计如何验证？

## 代码审查检查

- 是否把客户端传入的用户、机构或租户 ID 当作授权依据。
- 是否拼接 SQL、命令、HTML 或跳转 URL。
- 是否记录完整请求、环境变量、Token 或敏感对象。
- 是否使用固定 IV、可预测 nonce、硬编码密钥或过时算法。
- 是否只在前端隐藏按钮，而服务端未执行相同权限校验。

## 验证证据

- 未授权、跨租户和对象遍历测试。
- 密钥扫描结果及配置变量名清单。
- 脱敏、XSS、CSRF、SQL 注入和重放场景的测试结果。
- 安全例外的责任方确认、理由和有效期。
